## Agent Design and Pandas Functionality Summary

This document outlines the design of the query processing agents within the application, with a special focus on the newly implemented Pandas Agent.

### Overall Agent Design

The backend query processing is architected around a modular agent system, accessible via the `/query` endpoint. This endpoint accepts a natural language query and an `agent_type` parameter, which determines how the query is processed. Currently, two agents are supported: `sql` and `pandas`.

**Core Workflow for `/query` Endpoint:**

1.  **Request Reception:** Receives JSON containing `naturalLanguageQuery`, `metadata` (describing the uploaded data), and `agent_type`.
2.  **Configuration Check:** Verifies OpenAI API key configuration.
3.  **Context Validation:** Ensures a file has been uploaded and table/data context is available.
4.  **Agent Dispatch:**
    *   If `agent_type` is "sql", it calls `handle_sql_agent_query`.
    *   If `agent_type` is "pandas", it calls `handle_pandas_agent_query`.
5.  **Response Formatting:** The chosen agent handler returns a structured dictionary, which the `/query` endpoint then maps to a standardized JSON response for the client. This response includes:
    *   `code_type`: ("sql" or "pandas")
    *   `generated_code` or `sql_query`: The code generated by the LLM.
    *   `results`: The data returned by executing the code (JSON format).
    *   `error`: Any error messages from code generation or execution.
    *   `natural_language_response`: An LLM-generated summary of the results or process.

**Prompt Generation:**
All prompts sent to the Language Model (LLM) are constructed using a standardized approach employing Python's f-strings and `textwrap.dedent`. This ensures prompts are clean, maintainable, and clearly structured. Prompts typically include:
*   Instructions for the LLM (role, task).
*   Schema of the data (table name, column names, and types for SQL; DataFrame variable name and column details for Pandas).
*   The user's natural language query.
*   Context-specific information (e.g., failed code and error messages for reflection, or data summaries for natural language responses).
*   A placeholder for the LLM to insert the generated code or text.

### SQL Agent (`handle_sql_agent_query`)

The SQL agent processes natural language queries by converting them into SQL queries for execution against uploaded data files (CSV, Parquet, SQLite) via DuckDB.

**Workflow:**

1.  **SQL Generation:**
    *   An LLM (e.g., `text-davinci-003`) is prompted with the data schema and user query to generate an initial SQL query.
2.  **SQL Execution:**
    *   The generated SQL is executed using `execute_duckdb_query`.
3.  **Error Reflection & Correction (if needed):**
    *   If the first SQL execution fails, the LLM is prompted again with the original query, schema, the failed SQL, and the error message. It's asked to provide a corrected SQL query.
    *   The corrected SQL is then executed.
4.  **Natural Language Summary:**
    *   If SQL execution is successful (either initially or after correction), the results are summarized.
    *   The LLM is prompted with the user query, the successful SQL query, and a text summary of the SQL results to generate a concise natural language answer.
5.  **Return Value:** A dictionary containing the final SQL query, JSON results, natural language summary, any errors, and the result DataFrame.

### Pandas Agent (`handle_pandas_agent_query`)

The Pandas Agent processes natural language queries by converting them into Python pandas code snippets for in-memory data manipulation.

**Workflow:**

1.  **Data Loading:**
    *   The agent first loads the previously uploaded data file (`current_uploaded_filepath`) into a pandas DataFrame (`input_df`).
    *   Supported file types: CSV, Parquet, SQLite.
    *   For SQLite, the `table_name` from the metadata is used to query the specific table into the DataFrame.
    *   Errors during data loading are caught and reported.
2.  **Pandas Code Generation (`generate_pandas_code_prompt` & LLM call):**
    *   An LLM is prompted to generate a Python pandas code snippet. The prompt includes:
        *   Instructions to act as a pandas code generation assistant.
        *   The DataFrame name to use in the code (hardcoded as `df`).
        *   The column names and inferred types from the `input_df`'s metadata.
        *   The user's natural language query.
        *   A directive that the final result of the pandas operations should be assigned to a variable named `result_df`.
3.  **Pandas Code Execution (`execute_pandas_code`):**
    *   The LLM-generated pandas code string is executed using `exec()`.
    *   The execution environment for `exec()` is carefully controlled:
        *   `globals`: Contains only `{'df': input_df.copy(), 'pd': pd}` (a copy of the input DataFrame and the pandas module).
        *   `locals`: A fresh dictionary to capture variables created by the executed code.
    *   After execution, the function attempts to retrieve `result_df` from the local scope.
    *   If `result_df` is a pandas Series, it's converted to a DataFrame.
    *   Handles various exceptions during execution (SyntaxError, NameError, KeyError, AttributeError, etc.) and returns appropriate error messages.
    *   If `result_df` is not found or is not a DataFrame/Series, an error is returned.
    *   **Security Note:** The use of `exec()` is acknowledged. While suitable for this controlled environment where the LLM is assumed to generate safe code, production systems would require robust sandboxing (e.g., restricted interpreters, containerization) for executing arbitrary code.
4.  **Natural Language Summary:**
    *   If pandas code execution is successful and `result_df` is available:
        *   The LLM is prompted with the user's original query, the generated pandas code, and a textual summary of the `result_df` (full DataFrame for <=5 rows, head + count otherwise).
        *   The LLM generates a concise natural language answer.
    *   If summarization fails, a fallback message is provided.
5.  **Return Value:** A dictionary containing the generated pandas code, JSON results (from `result_df`), natural language summary, and any errors.

This dual-agent approach allows the application to leverage the strengths of both SQL (for structured querying and larger datasets via DuckDB) and pandas (for flexible in-memory data manipulation and analysis) based on user needs or query complexity.
